# Investigation: FirebaseMessaging static init abort
## TLDR
This is a high-severity startup crash on Android: `Firebase.Messaging.FirebaseMessaging` throws a `TypeInitializationException` which leads to a SIGABRT. The code initializes Firebase Messaging before Firebase dependencies are confirmed/initialized, so on some devices the type initializer fails and the process aborts.
## Risk Assessment
High. Fatal crash on app start, ~1.2k occurrences across ~1.1k users, impacting production.
## Release Branch
v3.34.2(51)
## Release Version
3.34.2
## Error Summary
Sentry reports `Abort`/SIGABRT with breadcrumbs showing `System.TypeInitializationException: The type initializer for 'Firebase.Messaging.FirebaseMessaging' threw an exception.` This occurs right after early app initialization, during notifications setup.
## Root Cause
`EventStreaming.InitializeNotifications()` calls `InitializeFirebaseMessaging()` immediately, while `InitFirebase()` (which runs `FirebaseApp.CheckAndFixDependenciesAsync`) is kicked off but not awaited. `FirebaseMessaging` is accessed before Firebase dependencies are fully resolved/initialized, causing the static initializer to throw and the process to abort.
## Affected Code
- `Assets/Scripts/Events/EventStreaming.cs` (Firebase init + messaging registration order).
## Suggested Fix
Gate `InitializeFirebaseMessaging()` behind a successful Firebase dependency check/initialization. For example, await `FirebaseApp.CheckAndFixDependenciesAsync()` before registering `FirebaseMessaging.TokenReceived/MessageReceived`, or reuse `ExecuteWhenFirebaseReady()` to ensure `_isFirebaseInit` is true before touching `Firebase.Messaging.FirebaseMessaging`. Also log/skip initialization when dependencies are unavailable to avoid crashing.
