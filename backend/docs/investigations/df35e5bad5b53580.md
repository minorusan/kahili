# Investigation: Seasonal challenge duplicate milestone collection
## TLDR
This is a high-volume client error (1118 occurrences / 389 users) caused by the seasonal challenge flow issuing overlapping milestone collection requests. The MilestonesValidator detects that a milestone range was already successfully collected earlier in the session and logs an error instead of treating it as a no-op. The bug is in the client request/animation sequencing, not a backend crash.
## Risk Assessment
Moderate user-facing risk: reward collection requests can be attempted multiple times, and while the server likely rejects or ignores duplicates, the client still logs errors and may skip/loop milestone UI updates.
## Release Branch
v3.34.6
## Release Version
3.34.6
## Error Summary
The client logs `[Milestone Error] ... Trying to collect milestone, which was successfully collected already` from `SeasonalChallengeController.TryCollectAllRewards`, triggered by `MilestonesValidator.WasCollectedAlready`.
## Root Cause
`SeasonalChallengeController.TryCollectAllRewards` builds milestone ranges using `GetFromToMilestonesToCollect`, which only checks `_milestones[i].WasCollected`. After a successful server response, those milestones are not immediately marked collected before the next collection attempt can be queued (e.g., when `_updatedMilestones` remains non-empty and `UpdateMilestones` loops/re-enters). As a result, the next request overlaps the previously successful range, and `MilestonesValidator` flags it as already collected.
## Affected Code
- `Assets/Scripts/LiveOpsChallenges/SeasonalChallenge/SeasonalChallengeController.cs`
- `Assets/Scripts/LiveOpsChallenges/MilestonesValidator.cs`
- `Assets/Scripts/LiveOpsChallenges/BaseMultiChallengeController.cs`
## Suggested Fix
After a successful `CollectMilestoneReward` response, immediately advance local milestone state so subsequent requests do not overlap:
- Mark milestones in the collected range as collected or remove them from `_updatedMilestones` before any subsequent `TryCollectAllRewards` call.
- Alternatively (or additionally), update `GetFromToMilestonesToCollect` to skip milestones already covered by a successful `MilestonesValidator` request, or return early when `MilestonesValidator` reports `WasCollectedAlready` (treat it as a no-op rather than an error).
