# Investigation: UniTask Double-Await in Album Sprite Loading

## TLDR
**Medium-high severity.** When the album view is updated (via a DynamicUserState download completing), `AlbumViewSlot.Initialize()` fires a fire-and-forget `SetSpriteAsync` on a preserved `UniTask<Sprite>`. If the sprite hasn't finished loading when a second update arrives, `Initialize` is called again on the same slot, triggering a second concurrent await on the same UniTask — which UniTask's `Preserve()` does not support before completion. This causes an `InvalidOperationException` affecting 218 users with 1,287 occurrences. It likely disrupts album UI rendering for affected users.

## Risk Assessment
- **Severity**: Medium-high — causes visible UI errors and may leave album slot icons invisible (since `image.enabled` is set to `false` at the start of `SetSpriteAsync` and never restored on failure)
- **Scope**: 218 affected users, 1,287 occurrences over ~6 days
- **Status**: Escalating
- **Data loss risk**: None — this is a UI/display bug only

## Release Branch
`v3.34.6`

## Release Version
First seen in `3.34.2`

## Error Summary
- **Exception**: `InvalidOperationException: Already continuation registered, can not await twice or get Status after await.`
- **Location**: `ExtensionMethods.SetSpriteAsync` → awaiting a `UniTask<Sprite>` that already has a pending continuation
- **Trigger**: `AlbumController.UpdateAlbum()` → `AlbumView.UpdateView()` → `SetupScrollView()` → `AlbumViewSlot.Initialize()` — called while a previous `SetSpriteAsync` from an earlier `Initialize()` call is still awaiting the same sprite task

## Root Cause

The call chain is:

1. `AlbumContentController.InitializeAlbumSetData()` creates `AlbumSetViewData` objects with `IconTask = _spriteLoader.WaitForSprite(url).Preserve()` — sprite loading can take up to 60 seconds with retries (`AlbumSpriteLoader.cs`)
2. `AlbumView.Initialize()` calls `SetupScrollView()` which calls `AlbumViewSlot.Initialize()` for each slot
3. `AlbumViewSlot.Initialize()` fires `_icon.SetSpriteAsync(_data.IconTask).Forget()` — a fire-and-forget async operation that **awaits** the preserved `UniTask<Sprite>`
4. Before sprites finish loading, a `DynamicUserStateDownloader.GetCachedOrDownload` completion triggers `AlbumController.UpdateAlbum()` → `AlbumView.UpdateView()` → `SetupScrollView()` → `AlbumViewSlot.Initialize()` **again** on the same slot
5. The second `Initialize()` call fires another `SetSpriteAsync` that tries to **concurrently await** the same preserved `UniTask<Sprite>`

**The critical insight**: UniTask's `.Preserve()` only supports multiple **sequential** awaits (i.e., re-awaiting after the task has already completed and the result is cached). It does **not** support multiple **concurrent** awaits before completion, because the underlying `UniTaskCompletionSourceCore<T>` only allows a single continuation registration. The second concurrent await triggers the "Already continuation registered" error.

Additionally, because `SetSpriteAsync` sets `image.enabled = false` before the await, and the exception bypasses the `image.enabled = true` restoration (it only catches `OperationCanceledException`, not `InvalidOperationException`), the album slot icon likely remains invisible after this error.

## Affected Code

| File | Method | Role |
|------|--------|------|
| `Assets/Scripts/Album/AlbumViewSlot.cs` | `Initialize()` | Fires `SetSpriteAsync` on every call without guarding against concurrent loads |
| `Assets/Scripts/Utils/Extensions/ExtensionMethods.cs:190` | `SetSpriteAsync()` | Awaits the UniTask; sets `image.enabled = false` before await but doesn't restore on `InvalidOperationException` |
| `Assets/Scripts/Album/AlbumView.cs` | `UpdateView()` / `SetupScrollView()` | Calls `Initialize()` (not `UpdateData()`) on existing slots, re-triggering sprite loads |
| `Assets/Scripts/Album/AlbumContentController.cs` | `InitializeAlbumSetData()` | Creates preserved `IconTask` — `.Preserve()` insufficient for concurrent awaits |

## Suggested Fix

**Primary fix — Guard against re-triggering sprite load in `AlbumViewSlot.Initialize()`:**

```csharp
// AlbumViewSlot.cs
private bool _spriteLoadTriggered;

public void Initialize(Data data)
{
    _data = data;
    _progressBar.Init(_data.CurrentAmount, _data.MaxAmount);
    _title.text = _data.Title;

    // Only trigger sprite loading once per slot lifecycle.
    // The icon doesn't change between updates, and UniTask.Preserve()
    // does not support concurrent awaits before completion.
    if (!_spriteLoadTriggered)
    {
        _spriteLoadTriggered = true;
        _icon.SetSpriteAsync(_data.IconTask).Forget();
    }

    _onClick = _data.OnClick;
    _key = _data.Key;
    _notification.SetActive(data.HasNotification);
    SetProgressBarIsCompleted();
}
```

**Secondary hardening — Add `InvalidOperationException` handling in `SetSpriteAsync`:**

```csharp
// ExtensionMethods.cs
public static async UniTaskVoid SetSpriteAsync(this Image image, UniTask<Sprite> spriteTask)
{
    if (image == null) return;
    image.enabled = false;
    try
    {
        var sprite = await spriteTask;
        if (image != null)
        {
            image.sprite = sprite;
            image.enabled = true;
        }
    }
    catch (OperationCanceledException)
    {
        Debug.LogWarning($"SetSpriteAsync was canceled on {(image == null || image.gameObject == null ? "Image is null" : image.gameObject.name)}");
    }
    catch (InvalidOperationException ex)
    {
        // Restore image visibility if double-await occurs
        if (image != null) image.enabled = true;
        Debug.LogWarning($"SetSpriteAsync failed on {(image == null || image.gameObject == null ? "Image is null" : image.gameObject.name)}: {ex.Message}");
    }
}
```

**Alternative approach**: Modify `AlbumView.UpdateView()` to use `UpdateData()` instead of `Initialize()` for existing slots, since icon sprites don't change between updates. This would require expanding `UpdateData()` to also update title and notification state.
