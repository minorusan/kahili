# Investigation: StageQuest popup Close null reference
## TLDR
This is a medium-severity client crash caused by `StageQuestPopupLogic.Close` assuming fields are always initialized. When the popup view is missing a matching stage set (or timer UI), `_stages` or `_timerComp` stays null and `Close()` dereferences them. Guarding cleanup and handling the missing-stage case prevents the crash.
## Risk Assessment
Medium: affects production users (27 occurrences) and can crash when the popup auto-closes or LiveOps completes.
## Release Branch
v3.34.7
## Release Version
3.34.6 (also seen since 3.33.3)
## Error Summary
`NullReferenceException` in `PlayPerfect.LiveOpsChallenges.MultiQuest.StageQuest.StageQuestPopupLogic+<Close>d__31.MoveNext` during popup close, often triggered by timer or LiveOps completion.
## Root Cause
`Close()` blindly uses `_stages` and `_timerComp` even when they were never initialized. `SetupView()` can early-return if no `StageSets` match the mission count, leaving `_stages` null; `SetupTime()` returns early if `TimerText` is null, leaving `_timerComp` null. When the timer callback or LiveOps completion closes the popup, the async `Close()` dereferences these fields and throws.
## Affected Code
- `Assets/Scripts/LiveOpsChallenges/MultiQuest/StageQuest/StageQuestPopupLogic.cs` (methods `SetupView`, `SetupTime`, `Close`)
## Suggested Fix
- Harden `Close()` to tolerate missing initialization:
  - `if (_stages != null) await _stages.Disappear();`
  - `if (_multiQuestController != null) { ... -= ...; }`
  - `if (_timerComp != null) _timerComp.Dispose();`
- If `SetupView()` fails to find a matching `StageSets`, either:
  - Close the popup immediately with a safe path (no stage/timer operations), or
  - Disable the close timer and avoid subscribing to LiveOps updates until initialization succeeds.
- Optionally add a guard in `ShowPopup()` to bail out before wiring handlers if `_stages` cannot be resolved.
