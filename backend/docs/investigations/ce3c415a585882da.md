# Investigation: Updating milestones failed (OperationCanceledException)
## TLDR
This is a low-severity, noisy error caused by expected cancellation during milestone updates. When the seasonal challenge flow is canceled (e.g., popup/operation cleared), the cancellation token aborts `WaitWhileUpdatingMilestone`, throwing `OperationCanceledException`, which gets logged as an error and sent to Sentry.

## Risk Assessment
Low: no evidence of data corruption; the error is triggered by normal control flow cancellation but pollutes error reporting and could mask real issues.

## Release Branch
v3.34.7

## Release Version
3.34.7

## Error Summary
`SeasonalChallengeController.UpdateMilestones` logs `Updating milestones failed System.OperationCanceledException: The operation was canceled.` during reward/milestone update flow. Stack shows the exception bubbling from `WaitWhileUpdatingMilestone` and being captured as an error.

## Root Cause
`UpdateMilestones` creates `_cancellationTokenSource` and calls `WaitWhileUpdatingMilestone`, which awaits `UniTask.WaitWhile(..., cancellationToken: _cancellationTokenSource.Token)`. When the seasonal challenge operation is cleared or reset, `Clear()` invokes `_cancellationTokenSource?.Cancel()`, causing `WaitWhileUpdatingMilestone` to throw `OperationCanceledException`. This exception is caught by the generic catch in `UpdateMilestones` and logged as an error, which Sentry records. The cancellation is expected behavior, not a failure, but it is treated as an error.

## Affected Code
- `Assets/Scripts/LiveOpsChallenges/SeasonalChallenge/SeasonalChallengeController.cs` (methods `UpdateMilestones`, `WaitWhileUpdatingMilestone`, `Clear`)

## Suggested Fix
Handle expected cancellations explicitly so they don’t produce error logs or Sentry issues. For example:
- Catch `OperationCanceledException` inside `UpdateMilestones` and return quietly when `_cancellationTokenSource.IsCancellationRequested` is true.
- Alternatively, change `WaitWhileUpdatingMilestone` to check `cancellationToken.IsCancellationRequested` and return without throwing (or call `UniTask.WaitWhile` without cancellation, then gate by `_isUpdatingMilestone`).
- Optionally reset `_isUpdatingMilestone`/`_isCollectingMilestones` on cancellation to keep state consistent.
