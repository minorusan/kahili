# Investigation: AvatarController NRE when setting sprite
## TLDR
This is a high-volume UI error that likely doesn’t crash, but it spams Sentry and can leave avatars blank. The async cached-avatar path sets `Image.sprite` after the controller or `Image` has been destroyed or is unset, and there’s no guard in `SetAvatarSprite`, so Unity throws a `NullReferenceException` inside `Image.set_sprite`.
## Risk Assessment
High frequency (3k+ events, 1.6k users) with low functional impact; risk is ongoing log noise, potential UI flicker, and wasted main-thread work during avatar updates.
## Release Branch
v3.34.7
## Release Version
3.34.7
## Error Summary
`System.NullReferenceException` in `UnityEngine.UI.Image.set_sprite` called from `ninja.Components.AvatarController.LoadRemoteAvatar`, after sprite retrieval from `SpriteCache`.
## Root Cause
`AvatarController.LoadRemoteAvatar`’s cached-avatar branch (`SpriteCache.IsImageCached`) awaits disk load and then calls `SetAvatarSprite` without checking `_avatarImage` or `_isDestroyed`. `SetAvatarSprite` itself does not guard `_avatarImage` before `_avatarImage.sprite = newAvatar`, so if the controller is destroyed/disabled during the async await (or `_avatarImage` is not assigned), Unity’s `Image.set_sprite` path throws `NullReferenceException`.
## Affected Code
- `Assets/Scripts/MainScreen/Avatar/AvatarController.cs:135` (`SetAvatarSprite` assigns `_avatarImage.sprite` without null guard)
- `Assets/Scripts/MainScreen/Avatar/AvatarController.cs:210` (cached branch calls `SetAvatarSprite` without `_isDestroyed/_avatarImage` guard)
## Suggested Fix
Add a centralized guard in `SetAvatarSprite` (and optionally in the cached branch) to avoid setting sprites on destroyed/unassigned `Image` components. For example: early-return if `_isDestroyed || !_avatarImage || !isActiveAndEnabled`, and optionally log once if `_avatarImage` is missing. Mirror the existing remote-branch guard in the cached path and in `LoadLocalDbAvatar` so async or late calls don’t hit Unity UI after teardown.
