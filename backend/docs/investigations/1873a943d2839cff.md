# Investigation: WinnersCup "Not initialized" on HandleLiveOpsSelectedAsync

## Release Branch
`origin/release/3.34.6`

## Release Version
Affects **3.33.3** and **3.34.0** (107 occurrences, 107 unique users, all iOS, first seen Feb 9 2026)

## Error Summary
When the LiveOps system tries to present a WinnersCup tournament to the user (e.g. entering lobby after a Seasonal Challenge popup closes), the call reaches `WinnersCupTournament.HandleLiveOpsSelectedAsync` which hits the `_isInitialized == false` guard and logs:
> `[WinnersCup] Can't enter selected flow. Not initialized`

The tournament object exists in the manager's dictionary but was never successfully initialized.

## Root Cause
**A "zombie" tournament is left in the `_winnersCups` dictionary when initialization fails.**

In `WinnersCupManager.RegisterLiveOp()` (`WinnersCupManager.cs`):
```csharp
bool IRegisterToLiveOpsService.RegisterLiveOp(DynamicInAppOperation operation, ...)
{
    if (_winnersCups.ContainsKey(operation.Id))
        return false;

    var tournament = _tournamentFactory.Create(operation);
    _winnersCups[operation.Id] = tournament;           // <-- Added BEFORE init
    _allRegisteredCups.Add(...);
    _dynamicInAppOperations[operation.Id] = operation;  // <-- Added BEFORE init
    return tournament.Initialize();                     // <-- Can return false!
}
```

The tournament is inserted into `_winnersCups` and `_dynamicInAppOperations` **before** `Initialize()` is called. If `Initialize()` fails (returns `false`), the uninitialized tournament remains in both dictionaries. `Initialize()` fails when:
- The tournament operation's `EndTime` is null or already past
- The leaderboard data is null AND the join window has expired

Later, when the LiveOps system iterates operations (e.g. user closes a Seasonal Challenge popup → LiveOpsPresenter iterates to the next operation → finds WinnersCup), `WinnersCupManager.HandleShowLiveOpAsync` looks up the tournament by ID in `_winnersCups`, **finds the zombie**, and calls `HandleLiveOpsSelectedAsync` on it — which immediately fails with the "Not initialized" guard.

The `RemoveAfterDelayAsync()` background cleanup task (which would eventually remove expired tournaments) is never started because it runs only after a successful `Initialize()`.

## Affected Code
- **`WinnersCupManager.cs`** — `RegisterLiveOp()` (lines around 78-90): adds tournament to dict before checking init result
- **`WinnersCupTournament.cs`** — `Initialize()` (line ~229): returns `false` on failure, `HandleLiveOpsSelectedAsync()` (line ~537): logs the error

## Suggested Fix
Clean up the dictionaries when `Initialize()` fails. In `WinnersCupManager.RegisterLiveOp()`:

```csharp
bool IRegisterToLiveOpsService.RegisterLiveOp(DynamicInAppOperation operation, TimeSpan? timeLeft, LiveOpInitUserState userState, bool isMissionAdded)
{
    if (_winnersCups.ContainsKey(operation.Id))
        return false;

    var tournament = _tournamentFactory.Create(operation);
    _winnersCups[operation.Id] = tournament;
    _allRegisteredCups.Add($"ID: {operation.Id}, Name {operation.Name}");
    _dynamicInAppOperations[operation.Id] = operation;

    if (!tournament.Initialize())
    {
        // Clean up the zombie entry so it doesn't get presented later
        _winnersCups.Remove(operation.Id);
        _dynamicInAppOperations.Remove(operation.Id);
        return false;
    }

    return true;
}
```

## Risk Assessment

### Severity: **Medium** — Feature-breaking but not user-locking

**The user is NOT locked.** Here's why:
- `WinnersCupManager.HandleShowLiveOpAsync` calls `showNextCallback.Invoke(null)` **unconditionally**, so the LiveOps presentation flow always continues to the next operation. The user can proceed through other LiveOps and return to the lobby normally.
- The error returns `default(LiveOpsSelectResult)` which is gracefully handled.

**However, the WinnersCup feature is broken for affected users:**
- The tournament icon may appear in the lobby (if the slot view was partially set up) but tapping it does nothing.
- The error fires **every time** the LiveOps system tries to present WinnersCup to the user (e.g., each lobby entry), producing repeated Sentry reports.
- The zombie tournament persists until the user changes accounts or the LiveOps service resets.
- **Paying users are affected** — at least one event came from a payer (LTV=$38, Seniority=151 days), meaning this impacts engaged users who might participate in WinnersCup tournaments.

### Fix risk: **Low**
The fix is a simple cleanup of dictionary entries on init failure. The only behavioral change is that the uninitialized tournament is removed instead of lingering — which means the LiveOps system will skip over it instead of trying and failing. No new code paths are introduced.
