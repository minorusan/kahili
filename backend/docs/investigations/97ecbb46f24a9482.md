# Investigation: ContinuousOperationsManager rejects non-active popup parent
## TLDR
High-volume error log (64k+ occurrences, 62k+ users) but likely not a crash: popups still enqueue and display, yet parent/child sequencing is inconsistent. Root cause is LiveOps popup chaining passing a parent operation that is no longer the active continuous operation when other UI operations become active, so `SuspendOperation` logs an error. Fix by verifying the parent is the active operation before suspending; otherwise enqueue as a standalone operation.
## Risk Assessment
Moderate: no hard crash observed, but popup ordering/visibility can be inconsistent when multiple UI operations overlap; error noise is high and can mask real issues.
## Release Branch
v3.34.6
## Release Version
3.34.6 (first seen 3.34.0, continues through 3.34.7)
## Error Summary
`[ContinuousOperation] Non active operation child add not supported` and `[ContinuousOperation] No ongoing operations` from `ContinuousOperationsManager.SuspendOperation`, triggered while showing Personal Offers and other LiveOps popups. Active operation is often `SpecialOfferAnimationOperation` or `ToastTrophy` while the “current” (parent) is `ChallengePopupLogic` or other popups.
## Root Cause
LiveOps flow passes the previous popup as the parent for the next popup (`LiveOpsPresenter.IterateDynamicInApps` → presenter `HandleShowLiveOpAsync` → `PersonalOffersLogic.AsChild`). Between popup close and the next popup open, other continuous operations (unlock animations, toasts, winners cup) become active. When `PopupBase.OpenPopup` calls `_parent.SuspendWith(...)`, the manager rejects the request because the parent is not the active (last) operation or the queue is empty, logging an error. The child is then enqueued anyway, so the UI usually continues but the error spam persists.
## Affected Code
- `Assets/Scripts/Managers/ContinuousOperationsManager.cs` (SuspendOperation error path)
- `Assets/Scripts/Popups/PopupBase.cs` (OpenPopup → `_parent.SuspendWith`)
- `Assets/Scripts/LiveOps/LiveOpsPresenter.cs` (parent chaining between live ops)
- `Assets/Scripts/LiveOpsChallenges/Popups/PersonalOffersPopupPresenter.cs` (sets child relationship)
- `Assets/Scripts/Popups/ViewLogic/PersonalOffers/PersonalOffersLogic.cs` (AsChild)
## Suggested Fix
Guard parent suspension so it only happens when the parent is the current active operation:
- In `PopupBase.OpenPopup`, before calling `_parent.SuspendWith`, check whether the current manager implements `IContinuousOperationsUI` and whether `OngoingOperations` is non-empty and its last entry equals `_parent`. If not, call `EnqueueOperation(this)` and clear/ignore the parent.
- Alternatively, expose a `IsActive(IContinuousOperation)` / `TrySuspendWithActiveCheck(...)` API on `IContinuousOperationsManager` and use it in `PopupBase` and any LiveOps presenters.
This prevents “non-active”/“no ongoing operations” errors while preserving correct popup queuing.
