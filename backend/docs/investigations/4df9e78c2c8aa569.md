# Investigation: ContinuousOperation "No ongoing operations" after StampIt popup closes

## TLDR
**High severity** — 4,711 occurrences across 4,237 users. When a `ChallengePopupLogic` is suspended (hidden) by a `StampItPopupLogic`, the challenge popup's internal timer keeps running and can fire `Close()`, which dequeues it from the operations stack while StampIt is still on top. When StampIt later closes and the LiveOps iteration tries to show the next `ChallengePopupLogic` using the destroyed parent, `SuspendOperation` fails with "No ongoing operations" because the stack is empty. Additionally, there is a race condition in `ChallengePopupLogic.Close` that invokes `_onClose` (triggering the next popup to start opening async) BEFORE calling `await ClosePopup()` (which dequeues itself), causing the parent reference to become stale.

## Risk Assessment
- **Severity**: High — errors are escalating rapidly (massive spike on 2026-03-04 with 29 distinct child issues)
- **User Impact**: Logged as error-level. The error prevents the next popup in the LiveOps chain from opening correctly. Depending on downstream handling, users may miss challenge popups or see broken popup flow after closing StampIt. Not a crash, but likely degrades the LiveOps experience.
- **Breadth**: Affects both iOS and Android, across versions 3.29.5, 3.33.3, and 3.34.0

## Release Branch
`v3.33.3` (first seen release), also present in `3.34.0`

## Release Version
First seen: `3.33.3` — also reproduced in `3.29.5` and `3.34.0`

## Error Summary
Two related error messages from `ContinuousOperationsManager.SuspendOperation`:
1. **"No ongoing operations"** — the `_ongoingOperations` list is empty when trying to suspend
2. **"Non active operation child add not supported"** — the parent operation exists in the list but is not the last/active item

Both occur in the same call chain: `StampIt closes → LiveOpsPresenter.IterateDynamicInApps → ChallengesPresenter → ChallengePopupLogic.ShowPopup → PopupBase.OpenPopup → SuspendWith → SuspendOperation`

## Root Cause
There are two interacting bugs:

### Bug 1: Timer not paused when ChallengePopupLogic is suspended
In `ChallengePopupLogic.ShowPopup` (`Assets/Scripts/LiveOpsChallenges/Popups/Logic/ChallengePopupLogic.cs`), a timer is set up:
```csharp
_timerComp = new TimerComp(_timeProvider, _popupView.TimerText, TimeFormat.Format.Standard,
    () => Close(PopupTracker.ACTION_INTERNAL).Forget());
```

When another popup (like StampIt) opens on top, `ChallengePopupLogic` is suspended via `PopupBase.SuspendOperation()` → `Hide()`, which deactivates the GameObject. However, the **timer keeps running** (it runs on UniTask's PlayerLoop, not the GameObject's lifecycle). When the timer expires, `Close()` is called, which:
1. Calls `_onClose?.Invoke(null)` (null because `IsHidden == true`)
2. Calls `await ClosePopup()` → `DestroyPopup()` → `DequeueOperation(this)` — **removing the ChallengePopupLogic from the operations stack while StampIt is still on top**

This is visible in the operations log: StampIt is Added, then ChallengePopupLogic is Removed 6ms later (timer fires), then StampIt is Removed seconds later when the user closes it. At that point the stack is empty.

Note: Unlike `ChallengePopupLogic`, `StampItPopupLogic.Close` correctly calls `_timerComp?.Dispose()` at the top of its `Close` method, but neither popup pauses the timer on suspension.

### Bug 2: Race condition — `_onClose` invoked before `ClosePopup`
In `ChallengePopupLogic.Close`:
```csharp
private async UniTaskVoid Close(string action)
{
    _tracker.CloseAction = action;
    _onClose?.Invoke(IsHidden ? null : this);  // (1) Triggers next popup creation
    await ClosePopup();                         // (2) Dequeues THIS popup
}
```

When the popup is visible (not suspended) and closes normally, it passes `this` to `_onClose`. This triggers `ChallengesPresenter.OnPopupClose` → `showNextCallback.Invoke(this)` → next `IterateDynamicInApps` iteration, which starts opening the next popup **asynchronously** with `this` as parent. Meanwhile, `await ClosePopup()` runs concurrently and dequeues `this` from the operations list. By the time the next popup's `OpenPopup()` reaches `SuspendWith(parent, ...)`, the parent has been destroyed.

### Combined effect
When StampIt closes, `StampItChallengePresenter` calls `showNextCallback.Invoke(parentOperation)` where `parentOperation` is the original `ChallengePopupLogic` stored at creation time. This reference is **stale** — the ChallengePopupLogic was destroyed by its own timer (Bug 1) or race condition (Bug 2). The next `ChallengePopupLogic` tries to `SuspendWith` using this stale reference, finds an empty operations list, and triggers the error.

## Suggested Fix

### Fix 1 (Primary — Timer suspension): Override `SuspendOperation` in `ChallengePopupLogic`
```csharp
// In ChallengePopupLogic.cs
public override void SuspendOperation()
{
    _timerComp?.Dispose();  // Stop the timer when suspended
    base.SuspendOperation();
}
```
And re-create the timer in a `ProceedOperation` override (or `ShowHidden`):
```csharp
public override async UniTask ShowHidden(CancellationToken ct = default)
{
    await base.ShowHidden(ct);
    // Re-calculate remaining time and restart timer if needed
    if (_popupView != null && timeLeft > TimeSpan.Zero)
    {
        _timerComp = new TimerComp(_timeProvider, _popupView.TimerText, TimeFormat.Format.Standard,
            () => Close(PopupTracker.ACTION_INTERNAL).Forget());
        _timerComp.TimerProcessing(remainingTime);
    }
}
```

### Fix 2 (Race condition): Move `_onClose` after `ClosePopup`
In `ChallengePopupLogic.Close` (and consider same for `StampItPopupLogic.Close`):
```csharp
private async UniTaskVoid Close(string action)
{
    _tracker.CloseAction = action;
    await ClosePopup();                         // Dequeue first
    _onClose?.Invoke(null);                     // Then notify (pass null since popup is destroyed)
}
```
Note: This changes the contract — `_onClose` will always receive `null` since the popup is destroyed by the time the callback fires. `ChallengesPresenter.OnPopupClose` and `StampItChallengePresenter` need to handle this by using the original parent reference instead.

### Fix 3 (Defensive — StampItChallengePresenter): Validate parent before reuse
In `StampItChallengePresenter.HandleShowLiveOpAsync`, don't blindly reuse `parentOperation`:
```csharp
// After StampIt closes, get the current active operation from the manager
// instead of using the stale parentOperation reference
showNextCallback.Invoke(parentOperation);  // Consider passing null or validating
```
