# Investigation: NullReferenceException in popup Hide() async methods

## Release Branch
v3.34.6

## Release Version
Affects versions 3.30.15, 3.33.3, 3.34.0 (present across multiple releases)

## Error Summary
Two related `NullReferenceException` errors occurring in async `Hide()` methods of UI popup components:

1. **RewardBubbleHint.Hide()** (`ninja.Components.RewardBubbleHint+<Hide>d__20.MoveNext`) — 825 occurrences, 804 affected users
2. **ChallengeCriteriaPopup.Hide()** (`ninja.LiveOpsChallenges.Popups.Logic.ChallengeCriteriaPopup+<Hide>d__16.MoveNext`) — 1,617 occurrences, 1,589 affected users

Combined: **2,442 total occurrences**, **2,393 affected users**. First seen 2025-12-09, still active as of 2026-03-02.

## Root Cause
Both components share an identical bug pattern: **concurrent execution of the async `Hide()` method with no re-entrancy guard**.

The race condition occurs as follows:

1. `AutoHideTimer()` starts an async delay, then calls `Hide()`
2. While the delay is pending, the user touches the screen — `LateUpdate()` detects input, cancels the current `CancellationTokenSource`, creates a **new** one, and calls `Hide().Forget()`
3. The `AutoHideTimer`'s `Hide()` call may already be in-flight or about to start. Now **two concurrent `Hide()` executions** share the same `_cancellationTokenSource` field (which was just replaced)
4. The first `Hide()` to complete destroys or deactivates the `GameObject`
5. The second `Hide()` resumes after its `UniTask.Delay` and attempts to access `gameObject` (e.g., `gameObject.SetActive(false)` or `Destroy(gameObject)`) on an already-destroyed object — **NullReferenceException**

Additionally, in `RewardBubbleHint.LateUpdate()`, the pattern of cancel-then-recreate-then-call `Hide()` can fire on every frame as long as input is held, spawning multiple concurrent `Hide()` calls.

The `catch (OperationCanceledException)` block only handles cancellation — it does **not** catch `NullReferenceException`, so the exception propagates up the async call chain (visible in the nested stack trace: `Hide` -> `AutoHideTimer` -> `Hide` -> etc.).

## Affected Code
- `Assets/Scripts/RewardContainer/RewardBubbleHint.cs` — `Hide()` method (d__20), `AutoHideTimer()` method (d__19), `LateUpdate()`
- `Assets/Scripts/LiveOpsChallenges/Popups/Logic/ChallengeCriteriaPopup.cs` — `Hide()` method (d__16), `AutoHideTimer()` method (d__14), `LateUpdate()`, `Show()` method (d__15)

## Suggested Fix

### Fix for RewardBubbleHint.cs
Add a re-entrancy guard and a destroyed-object check:

```csharp
// Add field:
private bool _isHiding = false;

private async UniTask Hide()
{
    if (_isHiding) return;
    _isHiding = true;

    try
    {
        _canHide = false;

        if (_animator != null)
        {
            _animator.SetTrigger(HIDE_TRIGGER);
        }

        await UniTask.Delay((int)TimeSpan.FromSeconds(HIDE_ANIMATION_TIME).TotalMilliseconds, cancellationToken: _cancellationTokenSource.Token);

        if (this == null || gameObject == null) return;

        if (_hideInsteadOfDestroy)
            gameObject.SetActive(false);
        else
            Destroy(gameObject);
    }
    catch(OperationCanceledException)
    {
        _isHiding = false; // Allow re-entry after cancellation
    }
}
```

Also reset the guard in `LateUpdate` when starting a new hide cycle:
```csharp
// In LateUpdate, before calling Hide():
_isHiding = false;
```

### Fix for ChallengeCriteriaPopup.cs
Apply the same pattern:

```csharp
// Add field:
private bool _isHiding = false;

public async UniTask Hide()
{
    if (_isHiding) return;
    _isHiding = true;

    try
    {
        _canHide = false;

        if (_animator != null)
        {
            _animator.SetTrigger(HIDE_TRIGGER);
        }

        await UniTask.Delay((int)TimeSpan.FromSeconds(HIDE_ANIMATION_TIME).TotalMilliseconds, cancellationToken: _cancellationTokenSource.Token);

        if (this == null || gameObject == null) return;

        gameObject.SetActive(false);
    }
    catch (OperationCanceledException)
    {
        if (this != null && gameObject != null)
            gameObject.SetActive(false);
        _isHiding = false;
    }
}
```

Reset `_isHiding = false` in `Show()` before starting a new show/hide cycle.

## Risk Assessment
**Medium**. The fix is a defensive guard pattern that only prevents duplicate execution — it does not change any game logic or visual behavior. The popup will still show and hide correctly, just without concurrent `Hide()` calls stomping on each other. The `this == null` / `gameObject == null` checks add safety against use-after-destroy without affecting normal flow. Both affected components are UI-only (tooltips/popups), so even in the worst case a failed fix would only affect tooltip visibility, not game state or monetization flows.
