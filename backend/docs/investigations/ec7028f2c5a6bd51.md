# Investigation: Corrupted tournament result element
## TLDR
This is a client-side error log triggered when the app receives a tournament result entry with a missing `SessionState`. The client doesn’t crash, but it logs an error for each corrupted element, so the issue is noisy and persistent. The most likely root cause is server-side data corruption or malformed API responses from `LoadResults`/`GetTournamentResult`, not a client bug.
## Risk Assessment
Medium: no crash, but repeated error logging (840 occurrences, 269 users) can mask real issues and indicates data inconsistency that can also affect UX (missing results).
## Release Branch
v3.34.6
## Release Version
3.34.7 (observed in events), first seen in 3.30.12
## Error Summary
`TournamentResultsManager.AddOrUpdateResults` logs "Can not process corrupted tournament result element!" when an element of the results list is null or has a null `SessionState`.
## Root Cause
`LoadResults` or `GetTournamentResult` is returning a list containing at least one corrupted/partial `TournamentResult` (null or missing `SessionState`). The client code explicitly detects this and logs an error; it does not originate from a thrown exception. This strongly points to backend data corruption or API response inconsistencies rather than a pure client defect.
## Affected Code
- `Assets/Scripts/Managers/TournamentResultsManager.cs` (AddOrUpdateResults input validation and logging)
- `Assets/Scripts/Backend/ServerManager.cs` (LoadResults/GetTournamentResult response handling)
- `Assets/Scripts/Managers/TournamentResultsBackendService.cs` (client fetch path)
## Suggested Fix
- Backend: sanitize `LoadResults`/`GetTournamentResult` responses to never include null or partially populated `TournamentResult` entries; filter out invalid rows and fix data integrity in storage so `SessionState` is always present.
- Client (defensive): keep the null/SessionState guard, but downgrade to a warning and include identifiers (e.g., room id if available) and request context to aid server-side cleanup. Optionally, filter results list before passing into `AddOrUpdateResults` to avoid repeated logging on retries.
